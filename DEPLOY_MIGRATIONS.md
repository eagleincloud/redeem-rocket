## Migration Contents

This migration adds the following to your database:

### New Columns on `biz_users` table:
- `feature_preferences` (JSONB): Stores user's feature selections
  ```json
  {
    "product_catalog": true,
    "lead_management": false,
    "email_campaigns": false,
    "automation": false,
    "social_media": false
  }
  ```
- `onboarding_step` (INTEGER): Tracks current onboarding step (0-6)
- `onboarding_ai_data` (JSONB): Stores intermediate AI-generated suggestions
- `onboarding_done` (BOOLEAN): Whether onboarding is complete

### New `business_products` table:
Stores AI-generated and user-created products with fields:
- `id` (UUID): Primary key
- `business_id` (TEXT): Foreign key to `biz_users`
- `name` (VARCHAR): Product name
- `description` (TEXT): Product description
- `category` (VARCHAR): Product category
- `price` (NUMERIC): Product price
- `image_url` (TEXT): Product image URL
- `created_from_ai` (BOOLEAN): Whether generated by Claude
- `is_active` (BOOLEAN): Whether product is active
- `created_at` (TIMESTAMPTZ): Creation timestamp
- `updated_at` (TIMESTAMPTZ): Last update timestamp (auto-managed by trigger)

### Indexes:
- GIN index on `biz_users.feature_preferences` for fast JSONB queries
- Indexes on `business_products(business_id)` and `business_products(business_id, is_active)` for performance
- Index on `business_products(business_id, is_active)` for filtering active products

### RLS Policies:
Row-level security policies ensure users can only:
- View their own business's products
- Insert products for their own business
- Update their own business's products
- Delete their own business's products
- View and update their own feature preferences

### Triggers:
- Auto-update `business_products.updated_at` on every update


## Troubleshooting

### Migration fails with "column already exists"
- The migration uses `IF NOT EXISTS` clauses, so it's safe to re-run
- If you still get errors, the columns may exist under different names

### RLS policy errors
- Ensure `auth.users` table exists (it should by default in Supabase)
- Verify auth is set up correctly
- RLS policies check if `id = auth.uid()` which requires Postgres auth context

### Foreign key constraint error: "cannot be implemented / Key columns incompatible types"
- **Fixed in current version**: `business_id` is now UUID type
- If you see this error, make sure you're using the latest migration file
- The `business_id` must be UUID to match `biz_users(id)` which is UUID

### "No such table" errors
- Ensure `biz_users` table exists before running this migration
- This migration should run after basic auth setup is complete
